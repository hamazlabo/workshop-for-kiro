# spec 駆動開発を試してみる

## このセクションの目的

AI コーディングエージェント を使って本格的なアプリケーションを作成する際、「ワンショット」での入力だと上手く行かなかったり、後で問題が発生する事が多い。
IDE 版の Kiro には、本番運用に耐えられるアプリケーションを作る為の機能として「Spec Coding」というものが備わっている。
本項目では、Kiro CLI の Skills を使って、Spec Coding の基礎を学習していく。

## 4.1. Vibe Coding vs Spec Coding

AI を使った開発の進め方は、大きく 2 つのスタイルに分けられる。

**Vibe Coding（バイブコーディング）** は、「こんな感じで作って」と思いつくままに AI へ指示を出し、出てきたものを見て「もう少しこう」「次はこれ」と会話しながら進めていくスタイル。料理にたとえると、レシピを決めずに冷蔵庫の中身を見ながら即興で作るようなもの。手軽で早く、アイデアを形にして試すのに向いている。一方で、全体像を決めずに進むため、作っているうちに方向がブレたり、後から「結局どういう仕様だったか」が誰にも分からなくなりやすい。

**Spec Coding（スペックコーディング / 仕様駆動開発・SDD）** は、コードを書き始める前に「何を・なぜ・どう作るのか」を仕様書（spec）としてまとめ、その合意した設計図に沿って AI に実装させるスタイル。家を建てる前に設計図を引くのと同じで、最初にひと手間かかる代わりに、関係者の認識がそろい、後から見返しても判断の理由が残り、大きな手戻りを防げる。

ちょっとした試作や使い捨ての確認なら Vibe Coding が速い一方で、複数人で長く使うものや、機能が増えて複雑になるものほど Spec Coding の効果が大きくなる。

## 4.2. なぜ spec 駆動開発が必要なのか?

なぜワンショットのプロンプトだと上手く行かないのか？
それは、AI エージェントが持つ以下の特性による。

### 4.2.1. AI は会話が変わると前のことを忘れる

AI エージェントは、会話（セッション）が変わると、それまでのやり取りを基本的に覚えていない。
そのため、その場の指示だけで開発を進めると、前に決めた設計の意図や「こう書くというルール」をすぐに忘れ、毎回ちがう考え方で作ってしまう。
その結果、作りかけのアプリ全体で書き方やルールがバラバラになり、ちぐはぐになってしまう危険がある。

### 4.2.2. 部品をつなげるときのミスを防ぐ

ファイル 1 つ、機能 1 つだけを作るような小さな作業なら、その場の指示でもちゃんと動くものができる。
しかし、いくつもの部品・データの保管場所・外部サービスとの連携が組み合わさる複雑な機能では、それぞれ単体ではうまく作れていても、いざつなぎ合わせると噛み合わずに動かない、ということがよく起こる。
あらかじめ「部品どうしのつなぎ方のルール」を仕様として決めておくことで、組み合わせたときのミスを減らせる。

### 4.2.3. 「なぜそうしたか」を残して、開発を続けやすくする

多くの古いプロジェクトでは、「なぜこの部品（ライブラリ）を選んだのか」「なぜこの作りにしたのか」といった大事な判断の理由が文書に残されておらず、担当者の記憶だけに頼っている。
そのため、その人が辞めたとたんに中身が誰にも分からない「ブラックボックス」になってしまう問題が起こりがち。

作り始める前に、「何が必要か」「どう作るか」「どの案を選び、何を捨てたか」を文書として残しておけば、後から人が代わってもスムーズに開発を続けられる。

これらの課題を克服する為に、全ての作業内容を文章として作成して、それに従って開発する手法が注目される事になった。

## 4.3. 手順

### 4.3.1. Kiro を起動して、計画を始める

```txt
sdd-plan スキルを使って、AI チャットアプリを作るのを手伝って下さい。
```

Kiro の場合、使うスキルを明示した方が精度が上がる。
ディスカッションしながら、今回は以下のドキュメントを作成する。

- requirements.md (要件定義)
- design.md (設計書)
- plan.md (技術計画書)
- task.md (タスクリスト)

### 4.3.2. ドキュメントを元に、開発を始める

```txt
sdd-develop スキルを使って、開発を始めて下さい。
```

## 4.4. kiro の設定

このワークショップでは、`.kiro/` ディレクトリに Kiro の設定ファイルをあらかじめ用意してある。
大きく **エージェント（agents）**・**スキル（skills）**・**ステアリング（steering）** の 3 種類で構成される。

```
.kiro/
├── agents/      # どんな役割の AI を動かすかの定義
├── skills/      # 「この作業はこう進める」という手順書
└── steering/    # プロジェクト共通のルール・前提（全エージェントが参照）
```

### 4.4.1 agents/ — エージェントの定義

「どんな役割・権限の AI を、どのモデルで動かすか」を決めるファイル。

| ファイル | 役割 |
| --- | --- |
| `sdd.json` | spec 駆動開発の**司令塔**。`sdd-plan` で仕様を作り、`sdd-develop` で実装を進める。実装時は細かい作業を `sdd-worker` に振り分ける（オーケストレーター）。 |
| `sdd-worker.json` | 司令塔から振られた**1 つの作業だけを担当する作業員**。担当タスクの範囲内だけを実装し、終わったら結果を報告する。 |

> それぞれ使用するモデルを使い分けている（司令塔は判断力の高いモデル、作業員は実装向けのモデル）。役割を分けることで、大きな開発でも破綻しにくくなる。

### 4.4.2. skills/ — スキル（作業の進め方）の定義

「特定の作業をどんな手順で進めるか」をまとめた手順書。エージェントはこれに沿って動く。

| ファイル | 役割 |
| --- | --- |
| `sdd-plan/SKILL.md` | **仕様を作る**手順。ユーザと対話しながら requirements → design → plan → task の 4 ドキュメントを順番に作成する（勝手に決めず、必ずヒアリングする）。 |
| `sdd-develop/SKILL.md` | **仕様から実装する**手順。4 ドキュメントを読み込み、タスクを上から順に実装。完成後は公開 URL をユーザに返すところまで行う。 |

### 4.4.3. steering/ — ステアリング（共通ルール）の定義

プロジェクト全体の前提やルールを書いたファイル。すべてのエージェントが常に参照し、判断のブレを防ぐ。

| ファイル | 役割 |
| --- | --- |
| `product.md` | **何を作るのか**。ワークショップの目的と、作る 2 つのアプリ（テトリス / AI チャット）の概要。 |
| `structure.md` | **どこに置くのか**。ディレクトリ構成とファイル配置のルール（アプリは `workspace/` 配下、など）。 |
| `tech.md` | **どう作るのか**。使用する技術・ポート・各アプリの技術方針（Streamlit、レポートは ReportLab、など）。 |
